# Frontend Check List

## Scope
- [ ] Делать только клиентское мобильное приложение.
- [ ] Не реализовывать интерфейсы шефа и администратора в текущей поставке.
- [ ] Опираться на уже существующие XML-экраны и дорабатывать их.
- [ ] Добавить только те новые Activity и Fragment, которые действительно нужны по сценарию.

## Architecture
- [ ] Использовать Kotlin.
- [ ] Использовать MVVM как базовую архитектуру.
- [ ] Разделить UI, ViewModel и слой данных.
- [ ] Не помещать бизнес-логику в Activity и Fragment.
- [ ] Использовать Repository как точку доступа к данным.
- [ ] Применять Koin для dependency injection.
- [ ] Использовать Coroutines и Flow для асинхронного обмена данными.
- [ ] Не писать огромные классы на сотни или тысячи строк.
- [ ] Держать каждую ViewModel, entity и DTO в отдельном классе и отдельном файле.
- [ ] Разделять экраны, модели состояния, DTO и мапперы по разным файлам.

## UI and layout
- [ ] Оставить XML-верстку как основной формат UI.
- [ ] Доработать существующие layout-файлы вместо полной замены.
- [ ] Поддержать текущие Fragment и Activity, уже лежащие в `app/src/main/res/layout/`.
- [ ] Сохранить понятную навигацию между экранами.
- [ ] Добавить новые экраны только под реальные сценарии: авторизация, расписание, детали класса, бронирование, профиль, мои бронирования, уведомления.

## Screens and flow
- [ ] Реализовать экран регистрации.
- [ ] Реализовать экран входа.
- [ ] Реализовать главный экран со списком ближайших классов.
- [ ] Реализовать экран расписания с фильтрами.
- [ ] Реализовать экран деталей класса.
- [ ] Реализовать экран подтверждения бронирования.
- [ ] Реализовать экран моих бронирований.
- [ ] Реализовать экран профиля.
- [ ] Реализовать экран уведомлений.
- [ ] Сохранять навигацию через понятные и предсказуемые переходы.

## Domain constraints in UI
- [ ] Показывать только доступные классы на ближайшие 7 дней по умолчанию.
- [ ] Отображать свободные места у класса.
- [ ] Показывать шефа, программу, время, длительность и цену.
- [ ] Давать выбрать прокатный пакет при бронировании.
- [ ] Давать указать аллергии в профиле и при бронировании.
- [ ] Показывать статус бронирования и возможность отмены по правилам.
- [ ] Не давать редактировать уже выставленную оценку.
- [ ] Не показывать оценку как отзыв о классе, только как отзыв о шефе.

## State and data handling
- [ ] Использовать отдельные модели состояния для экранов.
- [ ] Обрабатывать loading, success, empty state и error state.
- [ ] Показать empty state, если классов на ближайшие дни нет.
- [ ] Корректно обрабатывать отказ бронирования при отсутствии мест.
- [ ] Корректно обрабатывать отмену класса студией.

## Navigation and auth
- [ ] До авторизации не открывать защищённые экраны.
- [ ] После логина переходить в основную часть приложения.
- [ ] Сохранять токен в безопасном хранилище.
- [ ] Поддерживать выход из аккаунта.

## Networking
- [ ] Использовать Retrofit и OkHttp для API.
- [ ] Поддерживать централизованную обработку ошибок сети.
- [ ] Логировать сетевые вызовы для отладки на стороне клиента при необходимости.
- [ ] Не выводить в логи чувствительные данные.

## Validation and UX
- [ ] Валидировать email, телефон и пароль на форме.
- [ ] Подсвечивать ошибки ввода до отправки запроса.
- [ ] Не давать отправить пустую или некорректную форму.
- [ ] Показывать понятные сообщения о недоступности слота, ошибке сети и отказе сервера.

## Docker and build
- [ ] Не требовать Docker для самого Android-приложения.
- [ ] Убедиться, что frontend не зависит от локальной ручной настройки backend.
- [ ] Использовать конфигурацию API через build config или окружение проекта.

## Testing
- [ ] Проверить ViewModel unit-тестами.
- [ ] Проверить навигационные сценарии.
- [ ] Проверить ключевые экраны на пустые и ошибочные состояния.
