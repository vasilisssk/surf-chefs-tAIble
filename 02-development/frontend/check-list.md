# Frontend Check List

## Scope
- [x] Делать только клиентское мобильное приложение.
- [x] Не реализовывать интерфейсы шефа и администратора в текущей поставке.
- [x] Опираться на уже существующие XML-экраны и дорабатывать их.
- [x] Добавить только те новые Activity и Fragment, которые действительно нужны по сценарию.

## Architecture
- [x] Использовать Kotlin.
- [x] Использовать MVVM как базовую архитектуру.
- [x] Разделить UI, ViewModel и слой данных.
- [x] Не помещать бизнес-логику в Activity и Fragment.
- [x] Использовать Repository как точку доступа к данным.
- [x] Применять Koin для dependency injection.
- [x] Использовать Coroutines и Flow для асинхронного обмена данными.
- [x] Не писать огромные классы на сотни или тысячи строк.
- [x] Держать каждую ViewModel, entity и DTO в отдельном классе и отдельном файле.
- [x] Разделять экраны, модели состояния, DTO и мапперы по разным файлам.

## UI and layout
- [x] Оставить XML-верстку как основной формат UI.
- [x] Доработать существующие layout-файлы вместо полной замены.
- [x] Поддержать текущие Fragment и Activity, уже лежащие в `app/src/main/res/layout/`.
- [x] Сохранить понятную навигацию между экранами.
- [x] Добавить новые экраны только под реальные сценарии: авторизация, расписание, детали класса, бронирование, профиль, мои бронирования, уведомления.

## Screens and flow
- [x] Реализовать экран регистрации.
- [x] Реализовать экран входа.
- [x] Реализовать главный экран со списком ближайших классов.
- [x] Реализовать экран расписания с фильтрами.
- [x] Реализовать экран деталей класса.
- [x] Реализовать экран подтверждения бронирования.
- [x] Реализовать экран моих бронирований.
- [x] Реализовать экран профиля.
- [x] Реализовать экран уведомлений.
- [x] Сохранять навигацию через понятные и предсказуемые переходы.

## Domain constraints in UI
- [x] Показывать только доступные классы на ближайшие 7 дней по умолчанию.
- [x] Отображать свободные места у класса.
- [x] Показывать шефа, программу, время, длительность и цену.
- [x] Давать выбрать прокатный пакет при бронировании.
- [x] Давать указать аллергии в профиле и при бронировании.
- [x] Показывать статус бронирования и возможность отмены по правилам.
- [x] Не давать редактировать уже выставленную оценку.
- [x] Не показывать оценку как отзыв о классе, только как отзыв о шефе.

## State and data handling
- [x] Использовать отдельные модели состояния для экранов.
- [x] Обрабатывать loading, success, empty state и error state.
- [x] Показать empty state, если классов на ближайшие дни нет.
- [x] Корректно обрабатывать отказ бронирования при отсутствии мест.
- [x] Корректно обрабатывать отмену класса студией.

## Navigation and auth
- [x] До авторизации не открывать защищённые экраны.
- [x] После логина переходить в основную часть приложения.
- [x] Сохранять токен в безопасном хранилище.
- [x] Поддерживать выход из аккаунта.

## Networking
- [x] Использовать Retrofit и OkHttp для API.
- [x] Поддерживать централизованную обработку ошибок сети.
- [x] Логировать сетевые вызовы для отладки на стороне клиента при необходимости.
- [x] Не выводить в логи чувствительные данные.

## Validation and UX
- [x] Валидировать email, телефон и пароль на форме.
- [x] Подсвечивать ошибки ввода до отправки запроса.
- [x] Не давать отправить пустую или некорректную форму.
- [x] Показывать понятные сообщения о недоступности слота, ошибке сети и отказе сервера.

## Docker and build
- [x] Не требовать Docker для самого Android-приложения.
- [x] Убедиться, что frontend не зависит от локальной ручной настройки backend.
- [x] Использовать конфигурацию API через build config или окружение проекта.

## Testing
- [x] Проверить ViewModel unit-тестами.
- [ ] Проверить навигационные сценарии.
- [x] Проверить ключевые экраны на пустые и ошибочные состояния.
