# Backend Check List

## Scope
- [ ] Реализовать backend как REST API для мобильного клиента.
- [ ] Не делать web-view слой и не строить backend вокруг MVC-страниц.
- [ ] Считать существующий backend отсутствующим, проектировать API и хранение данных с нуля.

## Domain and API boundaries
- [ ] Поддержать только клиентский сценарий в текущей поставке.
- [ ] Не включать в текущий скоуп интерфейсы шефа и администратора.
- [ ] Отдавать в клиент только данные, нужные для мобильного приложения.
- [ ] Не давать клиенту создавать или редактировать классы, шефов и расписание.
- [ ] Использовать контракт API как источник истины для сущностей и полей.

## Architecture
- [ ] Разделить код на слои: controller, service, repository, domain/entity, dto, mapper.
- [ ] Хранить бизнес-логику в service, а не в controller.
- [ ] Использовать DTO на границе API, не отдавать JPA-сущности наружу.
- [ ] Структурировать проект по фичам или доменным модулям.
- [ ] Не собирать контроллеры, сервисы и сущности в огромные файлы на сотни или тысячи строк.
- [ ] Держать каждую entity, service и controller в отдельном классе и отдельном файле.
- [ ] Выделять отдельные классы для DTO, request/response моделей и мапперов.

## Persistence
- [ ] Использовать PostgreSQL как основную БД.
- [ ] Описать схему через миграции Flyway.
- [ ] Зафиксировать статусы бронирований в БД и коде.
- [ ] Отразить связь клиента, класса, бронирования, шефа, проката и отзыва.

## Business rules from TЗ
- [ ] Запись на класс должна проверять наличие свободных мест.
- [ ] Должна быть защита от двойного бронирования одного места на один слот.
- [ ] Отмена бронирования без штрафа доступна только заранее, минимум за 24 часа.
- [ ] Поздняя отмена должна фиксировать штрафные точки.
- [ ] При отмене класса студией бронь не удаляется, а переводится в статус "отменен студией".
- [ ] Повторная запись на отменённый студией слот должна быть запрещена.
- [ ] Отзыв должен быть только о шефе, а не о классе.
- [ ] Отзыв нельзя изменить после сохранения.
- [ ] Оценка должна быть только в диапазоне 1-5.
- [ ] Постоянного клиента определять по количеству посещённых классов.

## Authentication and profile
- [ ] Поддержать регистрацию по имени, email, телефону и паролю.
- [ ] Поддержать вход по email или телефону и паролю.
- [ ] Не подключать social login на старте.
- [ ] Держать профиль клиента редактируемым только в разрешённых полях.
- [ ] Хранить аллергии как часть профиля клиента.

## Rental
- [ ] Поддержать выбор прокатного пакета при бронировании.
- [ ] Использовать только пакеты Basic, Standard, Premium.
- [ ] Учитывать стоимость проката в стоимости бронирования.
- [ ] Отражать доступность прокатных пакетов.

## Notifications and logging
- [ ] Поддержать push-уведомления и SMS как сценарий домена.
- [ ] Учитывать напоминания за 24 часа и за 2 часа до класса.
- [ ] Учитывать мгновенное уведомление при отмене класса студией.
- [ ] Добавить логирование входящих HTTP-запросов.
- [ ] Логировать статус, время выполнения и ошибки без утечки паролей и токенов.
- [ ] Маскировать чувствительные данные в логах.

## Security
- [ ] Использовать JWT для аутентификации.
- [ ] Хранить токены безопасно и проверять их валидность.
- [ ] Валидировать входные данные на уровне API.
- [ ] Возвращать понятные ошибки через единый формат ответа.

## Docker and delivery
- [ ] Подключить Docker для backend-сервиса.
- [ ] Поднять PostgreSQL через docker-compose для локальной разработки.
- [ ] Обеспечить воспроизводимый локальный запуск.

## Testing
- [ ] Покрыть бизнес-правила unit-тестами.
- [ ] Проверить критичные сценарии интеграционными тестами.
- [ ] Отдельно протестировать бронирование, отмену и создание отзыва.
