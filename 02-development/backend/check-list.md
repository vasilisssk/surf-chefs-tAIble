# Backend Check List

## Scope
- [x] Реализовать backend как REST API для мобильного клиента.
- [x] Не делать web-view слой и не строить backend вокруг MVC-страниц.
- [x] Считать существующий backend отсутствующим, проектировать API и хранение данных с нуля.

## Domain and API boundaries
- [x] Поддержать только клиентский сценарий в текущей поставке.
- [x] Не включать в текущий скоуп интерфейсы шефа и администратора.
- [x] Отдавать в клиент только данные, нужные для мобильного приложения.
- [x] Не давать клиенту создавать или редактировать классы, шефов и расписание.
- [x] Использовать контракт API как источник истины для сущностей и полей.

## Architecture
- [x] Разделить код на слои: controller, service, repository, domain/entity, dto, mapper.
- [x] Хранить бизнес-логику в service, а не в controller.
- [x] Использовать DTO на границе API, не отдавать JPA-сущности наружу.
- [x] Структурировать проект по фичам или доменным модулям.
- [x] Не собирать контроллеры, сервисы и сущности в огромные файлы на сотни или тысячи строк.
- [x] Держать каждую entity, service и controller в отдельном классе и отдельном файле.
- [x] Выделять отдельные классы для DTO, request/response моделей и мапперов.

## Persistence
- [x] Использовать PostgreSQL как основную БД.
- [x] Описать схему через миграции Flyway.
- [x] Зафиксировать статусы бронирований в БД и коде.
- [x] Отразить связь клиента, класса, бронирования, шефа, проката и отзыва.

## Business rules from TЗ
- [x] Запись на класс должна проверять наличие свободных мест.
- [x] Должна быть защита от двойного бронирования одного места на один слот.
- [x] Отмена бронирования без штрафа доступна только заранее, минимум за 24 часа.
- [x] Поздняя отмена должна фиксировать штрафные точки.
- [ ] При отмене класса студией бронь не удаляется, а переводится в статус "отменен студией".
- [ ] Повторная запись на отменённый студией слот должна быть запрещена.
- [x] Отзыв должен быть только о шефе, а не о классе.
- [x] Отзыв нельзя изменить после сохранения.
- [x] Оценка должна быть только в диапазоне 1-5.
- [ ] Постоянного клиента определять по количеству посещённых классов.

## Authentication and profile
- [x] Поддержать регистрацию по имени, email, телефону и паролю.
- [x] Поддержать вход по email или телефону и паролю.
- [x] Не подключать social login на старте.
- [x] Держать профиль клиента редактируемым только в разрешённых полях.
- [x] Хранить аллергии как часть профиля клиента.

## Rental
- [x] Поддержать выбор прокатного пакета при бронировании.
- [x] Использовать только пакеты Basic, Standard, Premium.
- [x] Учитывать стоимость проката в стоимости бронирования.
- [x] Отражать доступность прокатных пакетов.

## Notifications and logging
- [ ] Поддержать push-уведомления и SMS как сценарий домена.
- [ ] Учитывать напоминания за 24 часа и за 2 часа до класса.
- [ ] Учитывать мгновенное уведомление при отмене класса студией.
- [x] Добавить логирование входящих HTTP-запросов.
- [x] Логировать статус, время выполнения и ошибки без утечки паролей и токенов.
- [x] Маскировать чувствительные данные в логах.

## Security
- [x] Использовать JWT для аутентификации.
- [ ] Хранить токены безопасно и проверять их валидность.
- [x] Валидировать входные данные на уровне API.
- [x] Возвращать понятные ошибки через единый формат ответа.

## Docker and delivery
- [x] Подключить Docker для backend-сервиса.
- [x] Поднять PostgreSQL через docker-compose для локальной разработки.
- [x] Обеспечить воспроизводимый локальный запуск.

## Testing
- [x] Покрыть бизнес-правила unit-тестами.
- [ ] Проверить критичные сценарии интеграционными тестами.
- [ ] Отдельно протестировать бронирование, отмену и создание отзыва.
